# Transcendence

*This project has been created as part of the 42 curriculum by ecarvalh, lmaes, lvichi, rapcampo and rda-cunh.*

## Description

**Transcendence** is a full-stack web **marketplace** platform where users can buy and sell items with messaging, follow other users, leave product reviews, and browse listings through search, sorting, and pagination. The application is built around modern web frameworks, real-time communication through WebSockets, role-based access control, and a full observability stack.

The product is designed as a realistic, production-shaped application — not just a prototype that passes evaluation. We prioritised a secure core (authentication, roles, validated forms, HTTPS-only access), real-time interactions (chat with presence), and strong observability (Prometheus + Grafana with custom dashboards) over raw feature count.

**Key features**

- Email/password authentication and 42 OAuth 2.0 sign-in.
- User profiles with avatar upload, online-status badge, and a follow ("friends") system.
- Role-based admin area (ban/unban, promote/demote, listing moderation).
- Listings with images, reviews, and seller pages.
- Real-time 1-to-1 chat per listing, backed by WebSockets.
- In-app notification system for listing updates and deletions.
- Advanced search on listings with text filter, sorting, and pagination.
- Orders and payment-tracking endpoints (full purchase flow is **not** claimed as a module).
- Prometheus + Grafana monitoring of every container in the stack.
- Privacy Policy and Terms of Service pages reachable from the global footer.

## Instructions

### Prerequisites

- Linux host (tested on Debian / WSL2) with **Docker** and **Docker Compose v2**.
- **GNU Make**.
- ~4 GB of free RAM and ~5 GB of disk for images and volumes.

### Setup

1. Clone the repository.
2. Copy the example environment file and fill the values that are commented out or marked as secrets:
   ```bash
   cp .env.example .env
   ```
   Keys you will normally want to set yourself:
   - `FORTYTWO_CLIENT_ID`, `FORTYTWO_CLIENT_SECRET`, `FORTYTWO_REDIRECT_URI` — your own 42 OAuth application.
   - `DJANGO_SECRET_KEY`, `DATA_SERVICE_TOKEN`, `DB_*` passwords — replace the defaults before any non-local deployment.
   - `GRAFANA_ADMIN_USER` / `GRAFANA_ADMIN_PASSWORD` — credentials for the Grafana UI.
3. Bring the whole stack up with a single command:
   ```bash
   make
   ```
   To attach immediately to the aggregated container logs use `make dev`. Other targets: `make stop`, `make re` (rebuild), `make fclean` (down + volumes + images), `make database-backup`, `make database-restore`.
4. Open the app in the latest stable version of **Google Chrome** at:
   - `https://localhost` (the Nginx gateway exposes 443 with a self-signed certificate generated by the PKI service).

The certificate is self-signed, so the first visit asks you to accept it manually. All connections to the backend are forced through HTTPS, as required by the subject.

## Database Schema

The database is **MySQL 8** and is initialised by [`srcs/database/db/init/init-db.sql`](srcs/database/db/init/init-db.sql). The data service owns every query; no other container talks to MySQL directly.

### Tables

- **users** — `id`, `name`, `email` (unique), `password_hash`, `phone`, `role` (`User` / `Admin`), `status` (`Active` / `Suspended` / `Banned` / `Deactivated`), `avatar_url`, `created_at`, `updated_at`.
- **products** — `id`, `seller_id` → users, `name`, `slug` (unique), `description`, `price`, `images` (JSON), `status` (`Draft` / `Active` / `Paused` / `Deleted`), `avg_rating`, `review_count`, timestamps.
- **product_images** — `id`, `product_id` → products, `image_hash`, `display_order`, `created_at`.
- **orders** — `id`, `code` (unique), `buyer_id` → users, `status` (`Pending` / `Paid` / `Done` / `Cancelled` / `Refunded`), `subtotal`, `total`, `notes`, timestamps.
- **order_items** — `id`, `order_id` → orders, `product_id` → products, `seller_id` → users, denormalised `product_name`, `price`, `qty`, `subtotal`.
- **product_reviews** — `id`, `product_id`, `reviewer_id`, `order_items_id` (unique), `rating` 1–5, `title`, `body`, `status`, `created_at`.
- **payments** — `id`, `order_id` → orders, `provider`, `method`, `status`, `amount`, `paid_at`, `created_at`.
- **conversations** — `id`, `listing_id` → products, `buyer_id` → users, `seller_id` → users, `last_message`, `last_message_at`, `created_at`.
- **messages** — `id`, `conversation_id` → conversations, `sender_id` → users, `content`, `created_at`, `read_at`.
- **follows** — composite key (`user_id`, `following_id`), `created_at`, self-follow prevented by `CHECK`.
- **notifications** — `id`, `receiver_id` → users, `actor_id` → users, `type`, `product_id` → products, `payload` (JSON), `read_at`, `created_at`.

### Key relationships

- A **user** can be a seller of many **products** and the buyer of many **orders**.
- An **order** has many **order_items**, each pointing at a **product** and a **seller**; each item can carry one **review**.
- A **conversation** is uniquely defined by `(listing, buyer, seller)` and contains many **messages**.
- **follows** model the "friends" / following graph between users.
- **notifications** are emitted on listing create / update / delete and attach to a specific receiver.

### Web Major — Framework for frontend and backend

**Justification.** A marketplace requires a maintainable, scalable structure rather than ad-hoc scripts. We chose **React 18 + Vite** for the frontend (component model, fast HMR, strong ecosystem) and **Django 5 + DRF + Channels** for the backend (routing, authentication, ORM-free integration with the data service, and built-in support for WebSockets). Both qualify as frameworks under the subject's definition.

**Subject requirements**
- Use a frontend framework (React, Vue, Angular, Svelte, etc.).
- Use a backend framework (Express, NestJS, Django, Flask, Ruby on Rails, etc.).

**Owners.** All team members contributed to the foundation; ongoing maintenance: Erik (frontend), Raphael/Ricardo (Django backend), Leonardo Maes (data-service).

---

### Web Major — Real-time features

**Justification.** Buyers and sellers expect instant feedback: new messages, updated presence, and notifications must appear without a manual refresh. We centralised real-time behaviour in a single WebSocket layer powered by Django Channels and Daphne, backed by an in-memory channel layer (Redis-ready). Connection lifecycle, authentication, and broadcast are handled in [`api/consumers.py`](srcs/backend/app/api/consumers.py); presence ticks are tracked in [`api/presence.py`](srcs/backend/app/api/presence.py).

**Subject requirements**
- Real-time updates across clients.
- Handle connection / disconnection gracefully.
- Efficient message broadcasting.

**Owner.** Ricardo, Erik, Leonardo Maes.

---

### Web Major — User interaction (chat, profiles, friends)

**Justification.** This module collects three social pillars of the marketplace and integrates with everything else (auth, real-time, notifications). We implemented:

- **Chat** — 1-to-1 conversations scoped to a listing (`/api/chat/conversations/` and `/api/chat/conversations/{id}/messages/`). Live delivery uses the WebSocket consumer from Web Major; persistence is in MySQL.
- **Profiles** — public profile page per user (`/seller/:id`) with avatar, listing portfolio, follower/following counts, reviews, and a follow button.
- **Friends** — implemented as a **follow** graph (`follows` table) with endpoints under `/api/follow/` (action, following list, followers list, counts, personalised feed).

**Subject requirements**
- A basic chat system (send/receive messages between users).
- A profile system (view user information).
- A friends system (add/remove friends, see friends list).

**Owners.** Ricardo, Raphael, Erik, Leonardo Maes.

### User Major — Standard user management and authentication

**Justification.** Identity is the foundation of every other feature. We implemented secure registration and login with hashed/salted passwords (Django's PBKDF2 by default, JWT issued via `djangorestframework-simplejwt`), avatar upload, online status, a profile page, and the follow graph that satisfies the "friends" requirement.

**Subject requirements**
- Users can update their profile information. *(Profile page, `PATCH /api/auth/profile/`)*
- Users can upload an avatar (with a default avatar if none provided). *(Custom image service + initials fallback in `UserAvatar`.)*
- Users can add other users as friends and see their online status. *(`/api/follow/*` endpoints + presence-driven status dot.)*
- Users have a profile page displaying their information. *(`/profile` for self, `/seller/:id` for others.)*

**Owners.** Raphael, Ricardo, Erik.

### User Major — Advanced permission system

**Justification.** A marketplace has clearly differentiated roles: users, sellers, and administrators must not share the same capabilities. We enforce this both server-side (DRF permission classes — see [`api/permissions.py`](srcs/backend/app/api/permissions.py) and the `IsAdminRole` checks on every `admin_*` view) and client-side (`RoleGuard` and conditional menus). Admins can ban/unban users, promote/demote other admins, and access the listing moderation page.

**Subject requirements**
- View, edit, and delete users (CRUD). *(`/api/users/`, `/api/users/{id}/`, plus admin endpoints `/api/admin/bans/`, `/api/admin/manage/`.)*
- Roles management (admin, user, guest, moderator, etc.). *(Database enum `role` + status enum; promote/demote endpoints.)*
- Different views and actions based on user role. *(React `RoleGuard` and `/admin/*` routes; conditional menu items in `Header`.)*

**Owners.** Raphael, Leonardo Maes.

### User Minor — Remote authentication with OAuth 2.0

**Justification.** We implemented full **42 OAuth 2.0** sign-in: a `GET /api/auth/42/` redirect endpoint generates the state and builds the authorization URL, and `GET /api/auth/42/callback/` exchanges the code for a token, fetches the 42 profile, shadows the user locally, and issues JWTs. Successful and failed flows redirect to `/login/success` and `/login/error` respectively, handled by the `OAuthCallback` page on the frontend.

**Subject requirements**
- Implement remote authentication with OAuth 2.0 (Google, GitHub, 42, etc.).

**Owners.** Ricardo, Erik, Leonardo Vichi.
