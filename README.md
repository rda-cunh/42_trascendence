# Transcendence

*This project has been created as part of the 42 curriculum by ecarvalh, lmaes, lvichi, rapcampo and rda-cunh.*

## Description

**Transcendence** is a full-stack web **marketplace** platform where users can buy and sell items with real Stripe-backed payments, exchange messages in real time, follow other users, leave product reviews, and browse listings through search, sorting, and pagination. The application is built around modern web frameworks, WebSocket-driven real-time communication, role-based access control, internal CA-authenticated TLS on every service-to-service hop, and a full observability stack.

The product is designed as a realistic, production-shaped application — not just a prototype that passes evaluation. We prioritised a secure core (authentication, roles, validated forms, HTTPS-only access, internal PKI), real-time interactions (chat with presence), a complete purchase flow (Stripe Checkout → server-verified payment → persisted order), and strong observability (Prometheus + Grafana with custom dashboards) over raw feature count.

**Key features**

- Email/password authentication and 42 OAuth 2.0 sign-in.
- User profiles with avatar upload, online-status badge, and a follow ("friends") system.
- Role-based admin area (ban/unban, promote/demote, listing moderation) with an auto-provisioned admin account on first boot.
- Listings with image gallery, reviews, and seller pages.
- Real-time 1-to-1 chat per listing, backed by WebSockets.
- In-app notification system for listing creation, updates, and deletions.
- Advanced search on listings: text filter (debounced), category, price range, sort, and pagination.
- Full purchase flow: cart → Stripe Checkout → server-verified payment → persisted order → order confirmation in the UI.
- Internal PKI with a private CA and CA-signed certs on every internal service, so no service-to-service hop is plaintext.
- Prometheus + Grafana monitoring of every container in the stack, fronted by admin-gated access.
- Cross-browser compatibility: tested on Chrome, Firefox, and a third non-Chromium browser (see Accessibility module below).
- Privacy Policy and Terms of Service pages reachable from the global footer.

## Instructions

### Prerequisites

- Linux host (tested on Debian / WSL2) with **Docker** and **Docker Compose v2**.
- **GNU Make**.
- ~4 GB of free RAM and ~5 GB of disk for images and volumes.

### Setup

1. Clone the repository.
2. Copy the example environment file and fill the values that are commented out or marked as secrets:
   ```bash
   cp .env.example .env
   ```
   Keys you will normally want to set yourself:
   - `FORTYTWO_CLIENT_ID`, `FORTYTWO_CLIENT_SECRET`, `FORTYTWO_REDIRECT_URI` — your own 42 OAuth application.
   - `STRIPE_SECRET_KEY` — your Stripe test-mode secret key (the project uses Stripe Checkout in test mode). `STRIPE_SUCCESS_REDIRECT` and `STRIPE_FAIL_REDIRECT` are pre-filled with the gateway URLs and only need updating if the gateway host changes.
   - `DJANGO_SECRET_KEY`, `DATA_SERVICE_TOKEN`, `DB_*` passwords — replace the defaults before any non-local deployment.
   - `ADMIN_NAME` / `ADMIN_EMAIL` / `ADMIN_PASSWORD` — credentials for the bootstrap admin user, created automatically on first boot.
   - `GRAFANA_ADMIN_USER` / `GRAFANA_ADMIN_PASSWORD` — credentials for the Grafana UI.
3. Bring the whole stack up with a single command:
   ```bash
   make
   ```
   To attach immediately to the aggregated container logs use `make dev`. Other targets: `make stop`, `make re` (rebuild), `make fclean` (down + volumes + images), `make database-backup`, `make database-restore`.
4. Open the app at `https://localhost` (the Nginx gateway exposes 443 with a self-signed certificate generated by the PKI service). Supported browsers:
   - **Google Chrome** (latest stable) — the baseline.
   - **Mozilla Firefox** (latest stable).
   - **Midori** (or any WebKitGTK-based browser) — a third, non-Chromium engine, per the Accessibility module.

The certificate is self-signed, so the first visit asks you to accept it manually. All connections — both the browser-to-gateway edge and every internal service-to-service hop — are encrypted; the gateway verifies each upstream against the project's internal CA.

## Database Schema

The database is **MySQL 8** and is initialised by [`srcs/database/db/init/init-db.sql`](srcs/database/db/init/init-db.sql). The data service owns every query; no other container talks to MySQL directly.

### Tables

- **users** — `id`, `name`, `email` (unique), `password_hash`, `phone`, `role` (`User` / `Admin`), `status` (`Active` / `Suspended` / `Banned` / `Deactivated`), `avatar_url`, `created_at`, `updated_at`.
- **products** — `id`, `seller_id` → users, `name`, `slug` (unique), `description`, `price`, `images` (JSON), `status` (`Draft` / `Active` / `Paused` / `Deleted`), `avg_rating`, `review_count`, timestamps.
- **product_images** — `id`, `product_id` → products, `image_hash`, `display_order`, `created_at`.
- **orders** — `id`, `code` (unique), `buyer_id` → users, `status` (`Pending` / `Paid` / `Done` / `Cancelled` / `Refunded`), `subtotal`, `total`, `notes`, timestamps.
- **order_items** — `id`, `order_id` → orders, `product_id` → products, `seller_id` → users, denormalised `product_name`, `price`, `qty`, `subtotal`.
- **product_reviews** — `id`, `product_id`, `reviewer_id`, `order_items_id` (unique), `rating` 1–5, `title`, `body`, `status`, `created_at`.
- **payments** — `id`, `order_id` → orders, `provider`, `method`, `status`, `amount`, `paid_at`, `created_at`.
- **conversations** — `id`, `listing_id` → products, `buyer_id` → users, `seller_id` → users, `last_message`, `last_message_at`, `created_at`.
- **messages** — `id`, `conversation_id` → conversations, `sender_id` → users, `content`, `created_at`, `read_at`.
- **follows** — composite key (`user_id`, `following_id`), `created_at`, self-follow prevented by `CHECK`.
- **notifications** — `id`, `receiver_id` → users, `actor_id` → users, `type`, `product_id` → products, `payload` (JSON), `read_at`, `created_at`.

### Key relationships

- A **user** can be a seller of many **products** and the buyer of many **orders**.
- An **order** has many **order_items**, each pointing at a **product** and a **seller**; each item can carry one **review**.
- A **conversation** is uniquely defined by `(listing, buyer, seller)` and contains many **messages**.
- **follows** model the "friends" / following graph between users.
- **notifications** are emitted on listing create / update / delete and attach to a specific receiver.

### Web Major — User interaction (chat, profiles, friends)

**Justification.** This module collects three social pillars of the marketplace and integrates with everything else (auth, real-time, notifications). We implemented:

- **Chat** — 1-to-1 conversations scoped to a listing (`/api/chat/conversations/` and `/api/chat/conversations/{id}/messages/`). Live delivery uses the WebSocket consumer from Web Major; persistence is in MySQL.
- **Profiles** — public profile page per user (`/seller/:id`) with avatar, listing portfolio, follower/following counts, reviews, and a follow button.
- **Friends** — implemented as a **follow** graph (`follows` table) with endpoints under `/api/follow/` (action, following list, followers list, counts, personalised feed).

**Design note — follow graph instead of symmetric friendship.** We deliberately modelled the "friends" requirement as a **directional, many-to-many follow graph** rather than a symmetric, two-sided friendship. Both options cost roughly the same to implement (a single association table either way; the symmetric variant only adds a confirmation handshake), but a one-directional follow is a much better fit for a marketplace: buyers typically want to track sellers whose listings interest them without requiring the seller to accept and reciprocate, mirroring the pattern users already know from platforms like Etsy or eBay's *favourite seller*. As a side benefit, the same graph powers a personalised activity feed (`/api/follow/feed/`) without any extra modelling. We still expose `add` and `remove` actions and a "following" list, which is what the subject text actually requires.

**Subject requirements**
- A basic chat system (send/receive messages between users).
- A profile system (view user information).
- A friends system (add/remove friends, see friends list).

**Owners.** Ricardo, Raphael, Erik, Leonardo Maes.

### Web Minor — Notification system *(bonus)*

**Justification.** Per the subject's Bonus part (Chapter VII), modules beyond the required 14 points are eligible as bonus. We implemented a complete notification pipeline for listing creation, updates, and deletion. Notifications are persisted in the `notifications` table with a typed `payload`, and the frontend exposes a bell icon with an unread badge, a dropdown list, and mark-as-read / mark-all-read controls (`/api/notifications/*`).

**Design note — scope of "update" and "deletion" events.** The subject's wording mandates notifications for *creation, update, and deletion* actions, and we cover all three to be fully compliant. In a real marketplace, however, the genuinely useful event is the **creation** of a listing by a seller a user follows — that is what feeds the personalised activity stream. Notifying every interested user on every **update** (a price tweak, a description edit) or **deletion** (a listing taken down) would quickly produce notification fatigue and is rarely what users actually want; production marketplaces filter these aggressively (only price drops, only saved/favourited listings, etc.). We chose to keep the update/delete notifications in scope because the subject requires them and because they make the bonus claim defensible during evaluation, but we acknowledge this as a compliance-driven decision rather than a product-driven one.

**Subject requirements**
- A complete notification system for all creation, update, and deletion actions.

**Owner.** Ricardo.

### Module of Choice Major — Internal PKI / Authenticated TLS for all service-to-service traffic *(bonus)*

**Justification.** The subject only mandates HTTPS at the edge (browser ↔ gateway); all internal container-to-container traffic is allowed to be plaintext. We extend the trust chain **inside** the private network with our own certificate authority, so every hop on the path of a user request is encrypted and authenticated against a CA we operate. Internal services can no longer be impersonated by a rogue container, and internal traffic is no longer cleartext on the Docker network.

This is delivered as a **dedicated subsystem** — `srcs/pki/` — with its own compose service, generation script, SAN strategy, and integration changes in every other service folder. Cert generation is a hard dependency of the rest of the stack: the MySQL container, for example, is explicitly gated on cert generation completing first.

**What is shipped.**

- **Internal Certificate Authority.** `srcs/pki/scripts/gen-certs.sh` generates a 4096-bit RSA root with a 10-year validity (`CN=Transcendence Internal CA`), isolated from any public CA. The script is idempotent: if the CA already exists in the shared `/certs` volume it is preserved, so trust survives container restarts; only a deliberate volume wipe re-issues certificates.
- **Per-service leaf certificate with full SAN coverage.** A single internal leaf cert carries DNS SANs for every compose service that listens on TLS — `backend`, `database`, `gateway`, `data-service`, `image-service`, `prometheus`, `grafana`, `loki`, `promtail`, `mysqld-exporter`, `node-exporter`, `cadvisor`, `nginx-exporter-gateway`, `nginx-exporter-frontend`, `frontend` (15 entries).
- **Certificate distribution.** Certs are mounted into each container via a shared `/certs` volume; consumer services declare `depends_on` against the PKI generator so they only start after the cert is in place. Today's `fix(database): ensure MySQL starts after internal certificates are generated` cements this ordering.
- **TLS terminated by every internal service.** `backend`, `data-service`, `image-service`, Grafana, the gateway, and every exporter terminate TLS using the CA-signed leaf cert. Internal traffic is therefore encrypted end-to-end, not just at the edge.
- **Gateway verifies every upstream against the internal CA.** The Nginx gateway opens HTTPS connections to upstreams and verifies the upstream certificate against our CA on every route:
  ```nginx
  proxy_ssl_verify on;
  proxy_ssl_trusted_certificate /certs/${CA_CRT_NAME};
  ```
  Applied to `/api/`, `/api/auth/`, `/api/orders/`, `/ws/`, `/images/`, the frontend route, and the monitoring subpath. An upstream without a valid CA-signed certificate fails the handshake and the gateway returns a 502 — a rogue container that does not hold one of our private keys cannot pretend to be `backend:8000` or `data-service:9000`.

- Propose a feature outside the listed modules that materially improves the project's quality, security, or scalability.
- Implement it as a non-trivial, integrated piece of work.

### Module of Choice Minor — Stripe payment integration *(bonus)*

**Justification.** A marketplace needs a real payment path to be credible. We integrated **Stripe Checkout** with full server-side payment verification before persisting any order — not a "Stripe button," but a complete cart → payment → confirmed order loop that closes the e-commerce primitive the marketplace concept implies.

**What is shipped.**

- **Frontend flow.** A dedicated `features/checkout/` module with four pages: `Checkout` (cart review + initiation), `CheckoutRedirect` (hand-off to Stripe), `CheckoutSuccess` (return path), and `CheckoutFailure` (error path). The success page hands the `session_id` back to the backend for verification before showing confirmation.
- **Backend session creation.** `POST /api/orders/create-checkout/` (see [`api/views.py`](srcs/backend/app/api/views.py)) iterates the cart, fetches each listing's authoritative price from the data service (clients cannot tamper with prices), assembles Stripe `line_items` in cents, opens a Stripe Checkout Session with `mode="payment"`, and stamps `buyer_id` into the session metadata. The Stripe-hosted checkout URL is returned to the frontend.
- **Server-verified order persistence.** On the return path, the frontend calls `POST /api/orders/<session_id>/`. The backend retrieves the session from Stripe, verifies that `payment_status == "paid"`, and cross-checks the `buyer_id` stamped in the session metadata against the authenticated user before forwarding to the data service to persist the order. If verification fails (`paid` not set, or buyer mismatch), no order is created and the response is `403`.
- **Order surfacing.** Persisted orders appear immediately in the buyer's `/orders` page and in the admin dashboard.

**Design note — session retrieval vs webhook.** The standard one-shot Checkout pattern supports two persistence strategies: webhook delivery (`checkout.session.completed`) and session retrieval on the return URL. We use session retrieval because it is sufficient for the happy path and inspectable end-to-end during demo. A webhook receiver would add resilience against the user closing the tab between payment and the success redirect; we documented this in the planning notes as an optional follow-up.

### Accessibility Minor — Support for additional browsers *(bonus)*

**Justification.** The subject mandates compatibility with at least two **additional** browsers beyond Chrome. We tested and fixed UI/UX issues on **Firefox** (Gecko engine) and **Midori** (WebKitGTK engine), so the project is verified on three different layout engines: Blink (Chrome), Gecko (Firefox), and WebKit (Midori). Browsers sharing Chrome's Blink engine (Chromium, Brave, Vivaldi, Opera) are **not** claimed as the "additional" browsers, since the spirit of the requirement is engine diversity.

**Tested feature coverage.** Each browser was walked through the full flow:

- Authentication (email/password signup and login, 42 OAuth redirect + callback).
- Profile page (edit info, avatar upload with progress, password change).
- Listings (browse, search, filter, paginate, view detail, image gallery).
- Real-time chat (1-to-1 conversation, message send/receive, presence dot).
- Notifications (bell icon, unread badge, mark-as-read, mark-all-read).
- Stripe Checkout (redirect to Stripe, return, server-verified order persistence).
- Admin panel (user management, listing moderation, dashboard charts).

**Known limitations.** Documented per-browser so evaluators see the team has actually validated each browser rather than copy-pasting a claim:

- **Firefox.** Self-signed certificate must be accepted manually on first visit (same as Chrome). All claimed features work as on Chrome.
- **Midori (WebKitGTK).** Self-signed certificate flow is more terse than Chrome's. Stripe Checkout opens correctly because the checkout UI is rendered server-side by Stripe and does not rely on Blink-specific APIs. WebSocket-driven features (chat, presence, notifications) work over WebKitGTK's WebSocket implementation.
  - *Visual-only limitation:* the Three.js shader preview on listing cards may render without animated effects or fall back to a static thumbnail, depending on whether WebKitGTK's WebGL stack is enabled in the local build. This is cosmetic only — listings remain fully functional (browse, view, buy, chat). Three.js is not claimed as a module in this project, so the degradation does not affect any module being evaluated.
  - Older Midori builds (pre-9.x) used a different engine and are not supported; install a current Midori from the Debian repositories.

- Test all features in each browser and fix any browser-specific issues.
- Document any browser-specific limitations.
- Ensure consistent UI/UX across all supported browsers.
